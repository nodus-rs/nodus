# Pre-0.15.0 Follow-Ups

Status: Landed (pre-0.15.0 items complete; deferred items still tracked)
Date: 2026-05-13

## Summary

Three correctness and schema concerns flagged during the v0.14.0 → main review
need to land before 0.15.0 ships, because each one is awkward to fix
retroactively once the release is in users' hands:

- **(c)** `native_package_plugin_root` exists in two files with divergent
  fallback behavior — latent correctness bug.
- **(g)** `nodus.lock` carries both `managed_files` (legacy) and
  `managed_outputs` (new); shipping both fixes the public schema in a shape
  that takes a lockfile version bump to undo.
- **(e)** Codex marketplace registration must stay project-scoped so sync does
  not create stale global config entries.

Five other items (a, b, d, f, h) from the same review are pure internal
tidying and are explicitly deferred to 0.15.x or 0.16.0.

## Fundamental Facts

- v0.14.0 → main introduced native-plugin marketplaces under `.nodus/`, a new
  adapter-profile registry, activation hooks, root-skill packages, and
  managed-output provenance records (see `docs/specs/2026-05-09-*.md`).
- Codex supports local plugin marketplaces registered from project
  `.codex/config.toml`; Nodus writes the generated marketplace under
  `.agents/plugins/marketplace.json` and registers the project root as the local
  marketplace source.
- `LockedManagedOutput` (added in `693d648`) records adapter, kind, digest,
  origins per file. `Lockfile::managed_files` is the legacy paths-only list
  that pruning and ownership currently rely on.
- `native_package_plugin_root` exists in `src/adapters.rs` and
  `src/adapters/output.rs` with copy-pasted bodies; the `adapters.rs` copy
  additionally branches on `project_root_is_native_package_plugin_root`.

## Goals

- Eliminate the duplicate `native_package_plugin_root` so adapter writes and
  cleanup always agree on the plugin path. _(item c)_
- Pick a single source of truth for managed-file tracking in the lockfile
  shape that ships in 0.15.0. _(item g)_
- Make Codex marketplace registration automatic and repo-local. _(item e)_
- Record the deferred items so they don't fall on the floor between releases.

## Non-Goals

- Do not redesign `AdapterProfile`, hook helpers, or `OutputPlanOptions`
  threading in this milestone — those are the deferred refactors (a, b, d,
  f, h).
- Do not add migration tooling for users who already ran a `main` build that
  wrote global Codex config; leave existing global entries untouched.
- Do not introduce a new lockfile version unless (g) genuinely requires it;
  prefer additive/derived changes first.

## Items In Scope (Block 0.15.0)

### (c) Consolidate `native_package_plugin_root`

**Current state.** Two copies:

- `src/adapters.rs:613` — checks `PackageSource::Root`, also branches on
  `project_root_is_native_package_plugin_root` (handles the case where the
  project root *is* a native plugin tree).
- `src/adapters/output.rs:823` — checks `PackageSource::Root` only.

**Resolution.** Delete the `output.rs` copy; have `output.rs` call the
`adapters.rs` public function. Verify with `cargo test --workspace` that the
existing test suite passes — the divergence is a bug, not a feature, so any
test wiggle is signal.

**Acceptance.**
- Only one `native_package_plugin_root` function in the crate.
- `cargo test --workspace --all-features` green.
- `cargo clippy --workspace --all-targets --all-features -- -D warnings` green.

### (g) Revert unused `managed_outputs` lockfile schema

**Current state, on closer inspection.** `Lockfile.managed_outputs` was
**added but never populated** in production code:

- `LockedManagedOutput` and friends are defined in `src/lockfile.rs`.
- The field is initialized as `Vec::new()` everywhere
  (`Lockfile::new`, every doctor and resolver call site).
- The only references in non-test code are the type definition itself.
- Production tests use it as `managed_outputs: Vec::new()` or
  `lockfile.managed_outputs.clear()`.
- The spec at `docs/specs/2026-05-09-managed-output-provenance.md` is still
  in "Draft" status.

So we have a public lockfile field that is **always empty** for any 0.15.0
user. Shipping that is worse than not shipping it — third-party tooling
might start reading the field, and we'd have to keep it forever.

**Resolution.** Revert the unused schema. Specifically:

- Remove `Lockfile.managed_outputs`.
- Remove `LockedManagedOutput`, `LockedManagedOutputAdapter`,
  `LockedManagedOutputKind`, `LockedManagedOutputOrigin`.
- Remove the two managed_outputs-only tests and the `managed_outputs: Vec::new()`
  initializers throughout the suite.
- Keep the spec `docs/specs/2026-05-09-managed-output-provenance.md` as Draft
  — when it lands in 0.16.0, schema and producer ship together.

This is a strict reduction in public surface; no lockfile version bump
because the field was new in 0.15.0 and never written with content.

**Acceptance.**
- `LockedManagedOutput*` types and `managed_outputs` field gone from
  `src/lockfile.rs`.
- No `managed_outputs` references anywhere in `src/`.
- Lockfile round-trip tests still pass; the rest of the suite unchanged.
- `cargo test --workspace --all-features` green.

### (e) Move Codex registration to the local marketplace

**Current state.** Project sync no longer writes marketplace or plugin entries
to `~/.codex/config.toml` or `$CODEX_HOME/config.toml`. Codex registration is
project-local through `.codex/config.toml`, with the generated marketplace under
`.agents/plugins/marketplace.json` and plugin payloads under
`.nodus/packages/<alias>/codex-plugin/`.

**Resolution.** The user-config provenance track is superseded. Existing global
Codex config entries are left untouched, and `.codex/config.toml` is the project
surface for local marketplace registration, features, hooks, and direct project
settings.

**Acceptance.**
- Default `nodus sync` with the Codex adapter writes
  `.agents/plugins/marketplace.json` and registers that source from
  `.codex/config.toml`.
- Project sync leaves any supplied Codex user config path unchanged.
- README and `docs/hooks.md` document local marketplace registration.
- Unit tests cover local marketplace paths, source paths, ownership, doctor
  repair, and unchanged user config files.

## Deferred Items (Post-0.15.0)

Tracked here so they don't fall through the cracks. None of these affects
public file formats or user-facing behavior, so they can ride a later
release.

- **(a)** Collapse `OutputPlanOptions` + `to_lockfile_with_options` +
  `managed_paths_with_options` boolean-threading into a single `SyncContext`
  struct, or compute the booleans inside the resolver from `InstallPaths`.
- **(b)** Pull duplicated activation/hook helpers
  (`activation_script_stem`, `sanitized_script_segment`,
  `activation_script_contents`, `activation_hook_entry`,
  `managed_hook_command_from_relative_path`) out of `adapters/claude.rs` and
  `adapters/codex.rs` into a shared `adapters/hooks/` module.
- **(d)** Move residual `match adapter` arms into `AdapterProfile` fields:
  `plugin_dir_name`, `marketplace_relative_path`,
  `plugin_metadata_relative_path`.
- **(f)** Replace `adapter == Adapter::Codex` command special-case in
  `native_package_plugin_has_content` with a profile field
  (e.g. `commands_emit_as_skills: bool`).
- **(h)** `manifest/discover.rs::local_marketplace_source_is_generated_native_plugin`
  reverse-engineers adapter-layer layout via positional path components.
  Replace with a `strip_path_prefix(source, native_marketplace_root(...))`
  check exposed by the adapters layer.
